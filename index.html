Module 01

Introduction

What & Why of Parallel Programming

🤔

What is Parallel Programming?

ELI5

Imagine you have 1000 dishes to wash. Doing them one-by-one is sequential. Calling 4 friends to each wash 250 dishes at the same time is parallel programming.

Parallel programming divides a computation into independent subtasks that execute simultaneously across multiple processors or cores — reducing total wall-clock time. It's the answer to "how do we go faster when a single core can't go much faster?"

  • Rendering a 3D scene: divide the image into tiles, each core renders its tile independently
  • Sorting a billion numbers: split the array, sort each half in parallel, then merge
  • Web server handling 1000 requests: each request handled by a separate thread simultaneously
🔀

Concurrency vs Parallelism

ELI5

Concurrency = one barista taking orders from multiple customers, switching between them. Parallelism = multiple baristas, each serving their own customer at the same time.

These two concepts are often confused but are fundamentally distinct:

  • Concurrency is about dealing with multiple things at once — interleaving tasks by switching between them. A single-core CPU can be concurrent.
  • Parallelism is about doing multiple things at once — true simultaneous execution, requiring multiple cores or CPUs.

All parallel programs are concurrent, but not all concurrent programs are parallel.

  • Concurrency: A single-core OS running a browser + music player (fast context switching creates the illusion of simultaneity)
  • Parallelism: A 4-core CPU running 4 threads genuinely at the same time, each on its own core
  • Chef analogy: One chef (concurrency) = chops onions, stirs pot, checks oven in rotation. Two chefs (parallelism) = one chops, one stirs simultaneously.
🏠

Processes vs Threads

ELI5

A process is like a house — it has its own kitchen, bedroom, everything isolated. A thread is like a person in a shared apartment — multiple people share the same kitchen (memory) but each has their own room (stack).

  • Process: An independently running program instance with its own memory space, file handles, and resources. Isolated — a crash in one doesn't affect others. Context switching between processes is expensive.
  • Thread: A lightweight unit of execution within a process. Threads share the heap but each has its own stack and program counter. Cheaper to create and switch than processes.

Key implication: because threads share memory, they communicate easily — but they can also interfere with each other, leading to race conditions.

  • Chrome uses separate processes per tab for isolation — crashing one tab leaves others alive
  • A web server uses threads per request so they can share connection pools and caches in the same process
📈

Why It Matters

ELI5

For decades, every new CPU was just faster (higher clock speed). Then physics hit a wall. We can't keep making individual cores faster. Instead, we got more cores. Parallel programming is how software uses them all.

Moore's Law (transistor count doubling every ~2 years) continues, but clock speed stopped scaling around 2004 due to heat and power constraints. Modern CPUs have 4, 8, 16, or more cores.

A program that only uses one core leaves the other 15 idle. Without parallel programming, your 16-core machine runs most software no faster than a single-core machine from a decade ago.

  • Don't confuse concurrency with parallelism — a single-core CPU can be concurrent but never truly parallel.
  • Threads are lighter than processes, but sharing memory means you must manage synchronization carefully.
  • The OS scheduler decides when threads run — you don't control exact timing.
  • "More threads ≠ faster" — there's overhead to creating and switching threads. Understand the bottleneck first.
🧵
Module 02

Threads & Synchronization

The building blocks — and why they're tricky

👷

What is a Thread?

ELI5

A thread is like a worker in a factory. The factory (process) owns all the tools and materials. Each worker (thread) follows their own set of instructions independently, but uses the shared tools.

A thread is an independent sequence of instructions sharing memory with other threads in the same process. Each thread has its own:

  • Stack — local variables, method call frames
  • Program counter — which instruction it's currently executing

All threads in a process share: the heap (objects), static fields, and open file handles.

Thread t = new Thread(() -> {
    System.out.println("Thread ID: " + Thread.currentThread().getId());
});
t.start();  // start() ≠ run()! run() executes in the CURRENT thread
🔄

Thread Lifecycle

ELI5

A thread's life is like an employee's workday: hired (NEW) → clocked in, waiting for a task (RUNNABLE) → waiting for a colleague (BLOCKED/WAITING) → clocked out (TERMINATED).

NEW → RUNNABLE → BLOCKED / WAITING → TERMINATED
  • NEW: Thread object created, .start() not yet called
  • RUNNABLE: Eligible to run; OS scheduler decides when
  • BLOCKED: Waiting to acquire a monitor lock held by another thread
  • WAITING: Waiting indefinitely via wait(), join(), or park()
  • TIMED_WAITING: Waiting for a set duration — sleep(ms), wait(ms)
  • TERMINATED: Execution complete or exception thrown
🏎️

Race Conditions

ELI5

Two people editing the same Google Doc offline. Person A writes "Hello World", person B writes "Hi Planet". When they sync — one overwrites the other. The final result depends on who synced last, not intent.

A race condition occurs when the correctness of a program depends on the relative timing or interleaving of threads. With identical inputs, the program may produce different outputs on different runs. They're notoriously hard to reproduce and debug because they're timing-dependent.

Classic: two threads both doing counter++

  • Thread A reads counter: value = 5
  • Thread B reads counter: value = 5 ← same stale value!
  • Thread A writes 6
  • Thread B writes 6 ← should be 7, but overwrites A's result

Final: 6 instead of 7. One increment silently lost.

⚛️

Non-Atomic Operations

ELI5

counter += 1 looks like one step, but it's secretly three: read the value, add 1, write back. Another thread can barge in between any of these steps and corrupt the result.

At the bytecode level, this.value += delta compiles to multiple instructions:

  1. getfield — load current value from memory
  2. iload — load delta onto stack
  3. iadd — add them
  4. putfield — write result back to memory

A context switch can happen between any of these steps. An operation is atomic if it completes in a single, non-interruptible step. Java's AtomicInteger uses CPU-level compare-and-swap for true atomicity without locks.

  • int x = 5 — atomic (single write)
  • x++ — NOT atomic (read-modify-write, 3 steps)
  • AtomicInteger.incrementAndGet() — atomic via CAS (compare-and-swap)
🚪

Critical Sections & Mutual Exclusion

ELI5

A single-occupancy bathroom. Only one person can use it at a time. The door lock is mutual exclusion. The bathroom itself is the critical section.

A critical section is code that accesses shared mutable state and must not execute concurrently. Mutual exclusion (mutex) guarantees only one thread is inside at any given time.

Requirements for a correct mutex:

  • Safety: At most one thread in the critical section at a time
  • Liveness: Every thread that wants to enter eventually does
  • No unnecessary delay: Threads outside the critical section don't block others
🔒

Locks & synchronized

ELI5

synchronized is the lock on the bathroom door. When a thread enters, it acquires the lock. Others find it locked and wait. When the thread exits, it releases the lock and one waiter may enter.

In Java, every object has an intrinsic monitor lock. synchronized uses it:

  • Reentrant: A thread holding a lock can re-acquire it (prevents self-deadlock in recursive calls)
  • Automatic release: Lock is always released on block exit, even if an exception is thrown
  • Memory visibility: synchronized also ensures writes are visible to other threads
// Method form — locks 'this'
public synchronized void increment() { count++; }

// Block form — locks a specific object
synchronized(lock) { count++; }
  • Always call .start(), never .run() — .run() executes in the current thread, defeating the purpose.
  • Keep synchronized blocks as short as possible — long critical sections cause contention.
  • Synchronize on a private final object, not this — external code could also lock on this and interfere.
  • volatile only fixes visibility (ensures reads see latest writes) but does NOT make compound operations atomic.
⚠️
Module 03

Synchronization Hazards

What goes wrong when threads interact

🥢

Deadlock

ELI5

Two people at dinner, each holding one chopstick, each waiting for the other to put theirs down first. Neither will ever eat. They're stuck forever.

A deadlock occurs when two or more threads are each waiting for a resource held by another, creating a circular dependency. All four Coffman conditions must hold simultaneously:

  1. Mutual exclusion: Resources can't be shared
  2. Hold and wait: Thread holds one resource while waiting for another
  3. No preemption: Resources can only be released voluntarily
  4. Circular wait: A→B→C→A dependency chain

Prevention: always acquire multiple locks in the same global order across all threads.

// Thread 1: lock A then B
synchronized(lockA) { synchronized(lockB) { ... } }

// Thread 2: lock B then A  ← OPPOSITE ORDER = deadlock risk!
synchronized(lockB) { synchronized(lockA) { ... } }

// Fix: always acquire in the same order (A then B) everywhere
🚶

Livelock

ELI5

Two people walking toward each other in a narrow hallway. Both step right to let the other pass. Both step left. Both step right again... forever. They keep moving but make zero progress.

A livelock is like a deadlock, but threads are not blocked — they're actively running, just not making progress. Each thread responds to the other's state in a way that perpetually undoes the other's action.

Livelocks are harder to detect than deadlocks because the system appears busy (CPU is being used) but no useful work is done. Often caused by overly "polite" retry logic — add random backoff to break the symmetry.

🍽️

Starvation

ELI5

You're waiting in line at a restaurant but every time you're about to be served, a VIP customer cuts in front. You wait forever, even though the restaurant is running fine for everyone else.

Starvation occurs when a thread is perpetually denied access to a resource because other threads always get it first. The system as a whole makes progress — but the starving thread never does.

Common causes:

  • Unfair lock implementations (no FIFO ordering guarantee)
  • Thread priority starvation (high-priority threads monopolize CPU)
  • Overloaded thread pools (tasks never reach the front of the queue)

Java's ReentrantLock(true) uses a fair lock with FIFO ordering to prevent starvation.

🃏

Bad Interleavings

ELI5

Two chefs following the same recipe. Each step is fine alone. But if Chef A adds salt, then Chef B adds salt before either checks — the dish gets double the salt. Same steps, wrong order.

An interleaving is any possible ordering of instructions from multiple threads. A bad interleaving is one that produces incorrect results. The insidious part: most interleavings are fine — only a rare few are problematic, making bugs hard to reproduce.

Synchronization's job: restrict the set of possible interleavings to only the correct ones.

  • Thread A checks if (list.isEmpty()) → Thread B adds an element → Thread A calls list.get(0) thinking it's empty → crash
  • Thread A checks balance >= amount → Thread B withdraws → Thread A withdraws → overdraft
🍞

Producer-Consumer Pattern

ELI5

A baker (producer) puts loaves on a shelf. A cashier (consumer) takes them to sell. If the baker's too fast, the shelf overflows. If the cashier's too fast, they grab from an empty shelf. They need to coordinate.

The Producer-Consumer pattern decouples data production from consumption via a shared buffer. Critical invariants:

  • Consumer must wait if the buffer is empty
  • Producer must wait if the buffer is full

Naïve synchronized-only implementations lead to deadlock: the consumer waits while holding the lock, blocking the producer from adding items. Use wait()/notifyAll() — or better, Java's BlockingQueue which handles all coordination internally.

BlockingQueue<Item> queue = new LinkedBlockingQueue<>(capacity);

// Producer thread:
queue.put(item);   // blocks if full — no overflow possible

// Consumer thread:
Item item = queue.take(); // blocks if empty — no underflow possible
  • To prevent deadlock: always acquire multiple locks in the same global order across all threads — never opposite orders.
  • Livelock is often caused by over-eager retry logic — add random backoff (like Ethernet's CSMA/CD) to break the symmetry.
  • Prefer BlockingQueue over manual wait()/notify() coordination for producer-consumer.
  • Java's synchronized is not fair — there's no FIFO ordering. Use ReentrantLock(true) when fairness matters.
  • Deadlock detection: thread dumps (jstack) show which thread holds which locks and who's waiting for whom.

Module 04

Hardware Parallelism

Parallelism baked into silicon

🖥️

Von Neumann Architecture

ELI5

The CPU is the chef, RAM is the fridge, and the program is the recipe. The chef can only cook as fast as they can walk to the fridge and back to fetch ingredients. The bus between them is the bottleneck.

Modern computers follow the Von Neumann model: a single CPU, shared memory for both data and instructions, connected by a bus. This creates the Von Neumann bottleneck: the CPU is almost always faster than memory — it spends much of its time waiting for data. This fundamental tension motivates caches, out-of-order execution, and ultimately parallelism.

🗃️

Memory Hierarchy

ELI5

Registers = your pocket (tiny, instant). L1/L2 = your desk (small, fast). L3 = your filing cabinet (medium, decent). RAM = the library down the hall. Disk = a warehouse in another city.

  • Registers: ~1 cycle — on the CPU die itself
  • L1 cache: ~4 cycles — per-core, 32–64 KB
  • L2 cache: ~12 cycles — per-core, 256 KB – 1 MB
  • L3 cache: ~40 cycles — shared across cores, 4–32 MB
  • RAM: ~100–300 cycles — gigabytes of capacity
  • SSD: ~10,000 cycles — terabytes but very far away

A single cache miss costs 100–300 cycles — time the CPU could have executed ~100 instructions. Writing cache-friendly code (sequential access, small working sets) often beats algorithmic improvements.

📋

Cache Coherence

ELI5

Two chefs each keep a personal notepad (cache). If Chef A changes "2 eggs" to "3 eggs" on their notepad, Chef B's notepad still says "2 eggs". They're now working from inconsistent information.

In a multi-core CPU, each core has its own L1/L2 cache. When two cores cache the same address and one writes to it, the other's copy becomes stale. Cache coherence protocols (like MESI — Modified, Exclusive, Shared, Invalid) track state across all cores and automatically invalidate stale copies.

False sharing: two threads writing to different variables that happen to share a cache line cause constant invalidations — even though they don't logically share data. A major hidden source of slowdown.

🥔

Vectorization / SIMD

ELI5

Instead of peeling one potato at a time, use a machine that clamps 8 potatoes at once and peels all 8 in one motion. Same effort — 8× the throughput.

SIMD (Single Instruction, Multiple Data) applies one operation to multiple data elements simultaneously. A 256-bit vector register holds 8 floats (32-bit each) and can add all 8 pairs with a single instruction instead of 8 separate additions.

Compilers often auto-vectorize simple loops. Key requirement: each iteration must be independent — no data dependency between loop iterations.

⚡

Instruction-Level Parallelism (ILP)

ELI5

While the water is boiling, you can chop vegetables. The two tasks are independent — you don't need to wait for one before starting the other. A smart CPU does the same with instructions.

Modern CPUs execute multiple independent instructions per clock cycle:

  • Superscalar execution: Multiple execution units (ALU, FPU, load/store) running in parallel
  • Out-of-order execution: CPU reorders independent instructions to avoid stalls
  • Speculative execution: CPU predicts branch outcomes and executes ahead

ILP is managed entirely by hardware. But code structure matters: long dependency chains (output of step N feeds step N+1) limit ILP. Straight-line code and loop unrolling improve it.

🔧

Multi-core Processors

ELI5

Instead of making one chef impossibly fast, hire multiple chefs. Each has their own workstation but they share the pantry (RAM). A 4-core CPU is 4 chefs sharing one kitchen.

A multi-core processor integrates multiple complete CPU cores on a single chip. Each core independently executes threads. Cores share L3 cache and main memory but have private L1/L2 caches. The hardware cache coherence protocol keeps them consistent.

The number of threads that can truly run in parallel is bounded by the number of logical CPUs (including hyperthreading). More threads than cores means time-slicing, not true parallelism.

  • Access arrays sequentially — sequential access is cache-friendly (spatial locality). Random access causes cache misses.
  • Avoid false sharing: if two threads write to adjacent variables, pad them to be 64 bytes apart (one cache line), or use @Contended in Java.
  • On modern CPUs, synchronized compiles down to compare-and-swap (x86) or LL/SC (ARM) — hardware atomic instructions.
  • Vectorization requires loop bodies with no side effects, no function calls, and no data dependencies between iterations.
  • Hyperthreading shares physical core resources between 2 logical threads — useful for I/O-bound work, limited benefit for compute-bound work.

Module 05

Performance & Scalability

Measuring and reasoning about parallel speedup

📏

Key Metrics: T₁, T∞, Sₚ, Efficiency

ELI5

T₁ = one person does everything alone. T∞ = infinite people, but bottlenecked by tasks that must be sequential. Speedup = how much faster with p workers. Efficiency = are your workers actually busy, or standing around?

  • T₁ (Work): Total execution time on 1 processor — the sequential baseline
  • T∞ (Span): Time with infinite processors — limited by the longest dependency chain
  • Speedup Sₚ = T₁ / Tₚ: How much faster on p processors. Linear speedup (Sₚ = p) is ideal, rarely achieved.
  • Efficiency Eₚ = Sₚ / p: Fraction of time each processor does useful work. 1.0 = perfect; 0.5 = processors idle half the time.
🏋️

Work (T₁)

ELI5

Work is the total amount of "stuff" to do, regardless of how many workers you have. If you need to paint 100 fence posts, the total work is always 100 posts — dividing it among workers doesn't change the total, just how fast it gets done.

Work T₁ is the sum of all computation across all nodes in the task graph — the time a single processor would take to execute the entire program. It's independent of the number of processors.

Work gives a lower bound: Tₚ ≥ T₁/p — you can't go faster than if all processors worked with zero overhead.

Work Law Tₚ ≥ T₁ / p
📍

Span (T∞) — The Critical Path

ELI5

You're baking a cake: mix (5 min), bake (30 min), decorate (10 min). Even with 100 helpers, you can't finish in under 45 min — these steps must happen in order. That 45 min chain is the span.

Span T∞ is the length of the critical path — the longest chain of sequentially dependent tasks. It's the minimum possible execution time even with unlimited processors.

The ratio T₁/T∞ is the maximum useful parallelism — the most processors you could keep productively busy at once.

Span Law Tₚ ≥ T∞
🧱

Amdahl's Law

ELI5

You're moving apartments. Carrying boxes can be parallelized with helpers, but packing, driving, and unpacking are sequential. Even with 100 helpers for the carrying — the sequential parts still bottleneck you. That bottleneck is Amdahl's Law.

Amdahl's Law gives the maximum speedup for a fixed problem size:

Amdahl's LawSₚ ≤ 1 / (f + (1 − f) / p)

Where f = sequential fraction, p = processors. As p → ∞, speedup → 1/f.

  • 10% sequential code → max 10× speedup, ever
  • 5% sequential → max 20×
  • 1% sequential → max 100×

Example: 20% sequential, 8 cores: S₈ = 1/(0.2 + 0.8/8) = 3.33 (not 8). Optimize the sequential part first — it has disproportionate impact.

🚀

Gustafson's Law

ELI5

Amdahl asks: "same problem, more workers — how much faster?" Gustafson asks: "same time, more workers — how much bigger a problem can we solve?" With 8 workers you can tackle an 8× bigger job in the same time.

Gustafson's Law is the optimistic counterpart to Amdahl, for scaling the problem size:

Gustafson's LawSₚ = p − f · (p − 1)

As you add processors, you scale the problem to fill them. The sequential overhead stays constant while the parallel work grows — making the sequential fraction proportionally smaller. Real-world HPC workloads (weather simulation, protein folding) behave this way: more cores = bigger simulations in the same time.

Amdahl = strong scaling (fixed problem). Gustafson = weak scaling (bigger problem). Both matter.

🔀

Parallelism = T₁/T∞

ELI5

How many workers could you actually keep busy at once on this problem? That's the parallelism. If parallelism is 16, adding a 17th worker won't help — there's simply not enough independent work.

Parallelism = T₁/T∞ — the maximum average number of processors that can be productively employed simultaneously.

Maximum Useful ParallelismParallelism = T₁ / T∞

To increase parallelism: reduce span (shorten the critical path) and/or increase independent work. Sequential code that forces a dependency chain increases T∞ and shrinks parallelism.

  • Measure speedup relative to the best sequential algorithm, not the serialized version of your parallel code — otherwise the numbers look better than they are.
  • Profile to find the actual sequential fraction — it's almost always higher than you think (I/O, synchronization, JVM startup).
  • "Superlinear" speedup (Sₚ > p) is real — it can occur when the per-core working set fits in cache, while the sequential version thrashes the cache.
  • Efficiency below 0.5 usually signals too much synchronization overhead or poor load balancing.
  • Amdahl limits strong scaling. Gustafson enables weak scaling. For most scientific workloads, weak scaling is the relevant regime.

Divide & Conquer + Frameworks

Patterns for practical parallel computation

✂️

Divide & Conquer Parallelism

ELI5

You have 1024 papers to grade. Give 512 to a colleague, keep 512 yourself. Each of you splits again. Keep splitting until piles are small enough to grade directly. Then the grades bubble back up to a final total. Split = "divide", grade = "conquer", combine = "combine".

Divide and conquer is the fundamental pattern for parallel computation:

  1. Divide: Split the problem into two or more independent subproblems
  2. Conquer: Solve each subproblem recursively, in parallel
  3. Combine: Merge the results

It works because subproblems are independent — they don't share mutable state. The recursion tree has depth T∞ = O(log n) and total work T₁ = O(n). Works for: sorting, searching, reductions, matrix operations.

🏭

Thread Pools & ExecutorService

ELI5

Don't hire and fire workers for every single task — that's expensive. Instead, hire a fixed crew. When a task arrives, an available worker picks it up. When done, they grab the next task from the queue.

Creating a new OS thread for every small task is prohibitively expensive (~1ms per thread). Thread pools pre-create worker threads and maintain a task queue. Workers pull from the queue when idle.

  • Callable<T> — returns a value
  • Runnable — void, fire-and-forget
  • Future<T> — represents a pending result; .get() blocks until ready
ExecutorService pool = Executors.newFixedThreadPool(4);
Future<Integer> result = pool.submit(() -> expensiveCalculation());
Integer value = result.get(); // blocks until done
pool.shutdown();
🍕

Sequential Cutoff & Granularity

ELI5

You're slicing a pizza. Cutting it in half is efficient — two people eat simultaneously. But cutting each slice into 1000 crumbs and handing them to 1000 people is absurd — the coordination overhead swamps any benefit. There's a sweet spot.

Sequential cutoff: below a threshold problem size, stop recursing and solve sequentially. The overhead of creating/scheduling a task must be justified by the parallel work it unlocks. Rule of thumb: cutoff when N < 500–1000 for simple operations.

Granularity tradeoffs:

  • Too coarse: few large tasks → some cores starve (load imbalance)
  • Too fine: many tiny tasks → scheduling overhead dominates
🍴

ForkJoin Framework

ELI5

fork() = "start working on this subtask and I'll continue." join() = "wait until that subtask is done." Java's ForkJoin framework manages the underlying threads so you don't have to.

Java's ForkJoinPool (Java 7+) is designed for divide-and-conquer parallelism. Unlike ExecutorService, it doesn't deadlock on recursive task submission — it suspends waiting tasks rather than blocking threads.

  • RecursiveTask<V>: returns a value — override compute()
  • RecursiveAction: void — for in-place operations like sorting
class SumTask extends RecursiveTask<Long> {
    protected Long compute() {
        if (size <= CUTOFF) return sequentialSum();
        SumTask left = new SumTask(array, lo, mid);
        left.fork();             // async: start left half
        Long right = new SumTask(array, mid, hi).compute();
        return left.join() + right; // wait for left + combine
    }
}
ForkJoinPool pool = new ForkJoinPool();
long total = pool.invoke(new SumTask(array, 0, array.length));
🕸️

DAGs — Task Dependency Graphs

ELI5

Think of a recipe. Some steps can happen simultaneously (chop onions while water boils). Others must wait (add onions only after both are done). Drawing these dependencies gives you a DAG — a roadmap of what can run in parallel.

A Directed Acyclic Graph (DAG) represents parallel computation structure:

  • Nodes: individual tasks (with a cost/weight)
  • Edges: dependencies (B must wait for A → edge A→B)
  • Acyclic: no circular dependencies (otherwise: deadlock)

The critical path = the longest path through the DAG = T∞. Wider DAGs = more parallelism. The DAG is typically implicit and dynamic — it unfolds as the program runs.

🥷

Work-Stealing Scheduler

ELI5

Each worker has their own queue of tasks. When a worker finishes their queue, instead of idling, they secretly steal tasks from the back of a busy worker's queue. Everyone stays productive without a central coordinator.

Work-stealing is the scheduling algorithm used by ForkJoinPool. Each worker thread has a double-ended queue (deque):

  • Workers push/pop their own tasks from the front (LIFO — good cache locality)
  • Idle workers steal from the back of others' queues (FIFO — avoids contention)

This is self-balancing: highly recursive tasks stay local (cache-friendly), while load imbalance is corrected by stealing.

Graham's Theorem (1968)Tₚ ≤ T₁/p + T∞

RecursiveTask Pattern

ELI5

Write your parallel algorithm like a normal recursive function. Where you'd call yourself twice, fork() one call (runs in parallel) and directly execute the other. Then join() the forked one. That's the whole pattern.

Standard ForkJoin template — fork left, compute right directly in current thread, join left:

left.fork();                    // left runs asynchronously
V rightResult = right.compute(); // right runs in this thread
V leftResult  = left.join();    // wait for left, get result
return combine(leftResult, rightResult);

Why compute one directly? Halves thread creation. Why fork left first? Left may start running before right completes — more overlap opportunity. Always set a SEQUENTIAL_CUTOFF for base cases.

  • Fork the left subtask, compute the right one directly in the current thread — this halves thread creation while maintaining full parallelism.
  • Use RecursiveTask<V> when you need a return value, RecursiveAction for void tasks (like in-place sort).
  • Work-stealing makes unbalanced trees efficient — don't over-engineer task balancing; let the scheduler handle it.
  • Never block a ForkJoin thread on external I/O — use ManagedBlocker or a separate thread pool.
  • The ForkJoinPool.commonPool() is shared JVM-wide — avoid blocking tasks there.
  • Sequential cutoff is critical — without it, ForkJoin overhead makes parallel code slower than sequential for small inputs.

Parallel Algorithms

Classic patterns for parallel computation

🗺️

Maps — Embarrassingly Parallel

ELI5

You have 1000 photos to resize. Give 250 to each of 4 friends. There's no dependency between photos — each one can be processed completely independently. This is a map: apply the same operation to each element, no coordination needed.

A map applies a function independently to each element of a collection, producing a same-sized output. Since elements are independent, this is the easiest class of problem to parallelize — often called "embarrassingly parallel".

  • Work T₁ = O(n) — must process every element
  • Span T∞ = O(1) — with infinite processors, all elements process simultaneously
  • Parallelism = n — perfect scaling up to n processors

Examples: vector addition, image processing (blur, edge detection), encryption of independent records.

🧮

Reductions

ELI5

You want the total price of 1000 items. Split the list in half, give each half to someone, they sum their half, then you add the two partial sums. Keep splitting recursively — you get the answer in O(log n) time instead of O(n).

A reduction combines a collection into a single answer via an operator. The key requirement: the operator must be associative — the grouping of operations must not matter.

  • Work ✓: sum, max, min, product, AND, OR, XOR, count
  • Work ✗: subtraction (not associative), median (not combinable)

With divide-and-conquer: Work = O(n), Span = O(log n) — same total work, but the depth is now logarithmic.

// Parallel sum via divide-and-conquer
if (hi - lo <= CUTOFF) return sequentialSum(lo, hi);
SumTask left = new SumTask(lo, mid);
left.fork();
long right = new SumTask(mid, hi).compute();
return left.join() + right;  // associative combine
➕

Prefix Sum (Parallel Scan)

ELI5

You have scores [3, 1, 4, 1, 5]. You want the running totals: [3, 4, 8, 9, 14]. Seems sequential — each answer depends on the previous one. But with a clever two-pass tree trick, you can compute all running totals in O(log n) time.

Prefix sum (also called scan) computes output[i] = sum(input[0..i]). The parallel approach uses a binary tree in two passes:

  1. Up pass (reduce): Build the tree bottom-up, computing range sums at each node
  2. Down pass (distribute): Traverse top-down, propagating a "fromLeft" prefix value to each leaf

Result: Work = O(n), Span = O(log n) — transforms an apparently sequential algorithm into a parallel one. Prefix sum is a building block for many other parallel algorithms (sorting, filtering, histograms).

📦

Pack / Filter

ELI5

You have 1000 numbers and want only the even ones, in order. You can't just grab them independently — you need to know where each one goes in the output. Prefix sum tells you exactly that index, in parallel.

Filter/Pack keeps only elements satisfying a predicate, preserving order. It uses prefix sum as a sub-routine in 3 steps:

  1. Map: Create a bit-vector — 1 if the predicate holds, 0 otherwise
  2. Prefix sum: Compute running totals on the bit-vector → each 1's prefix sum value is its index in the output
  3. Map: Scatter elements to their computed output positions

Work = O(n), Span = O(log n) — a seemingly sequential operation made parallel by reducing it to prefix sum.

⚡

Parallel Quicksort

ELI5

Regular quicksort picks a pivot and partitions the array — that partition scan is sequential. To parallelize it, replace the sequential scan with two parallel pack operations: one gathers elements smaller than the pivot, one gathers larger. Then recurse on both halves in parallel.

Standard parallelization: fork both recursive calls. But the partition step itself (scanning the array to divide it) is still O(n) span — a bottleneck.

Optimized: replace the sequential partition with two parallel pack operations:

  • Pack elements < pivot into the left side
  • Pack elements > pivot into the right side
  • Place pivot; recurse both halves in parallel

Result: Span improves from O(n) to O(log²n) — the partition is no longer the bottleneck.

  • If an algorithm requires an associative operator, it can almost always be parallelized as a reduction. Check associativity first.
  • Prefix sum is a surprisingly powerful primitive — it's the backbone of filter, pack, histogram, stream compaction, and more.
  • Linked lists don't support efficient divide-and-conquer (O(n) to find the middle). Use arrays or balanced trees for parallel algorithms.
  • MapReduce (Hadoop, Spark) is the distributed extension of the map + reduction pattern — same ideas, different scale.
  • JVM warm-up affects benchmarks — always measure in a steady-state loop, not on the first invocation.

Java Memory Model

Why "obvious" code can silently break in parallel

🔀

Memory Reordering

ELI5

You write a note, leave it on the desk, and tell your colleague "check the desk." But your colleague looks before the note actually gets there — their local cache still shows the old version. Or your CPU decided to write the note later than you expected, as an optimization.

Modern systems reorder memory operations for performance, breaking the assumption that operations execute in the order you wrote them:

  • Compiler reordering: Optimizations preserve single-thread semantics but can break multi-thread assumptions (register hoisting, dead code elimination, instruction reordering)
  • Hardware reordering: Out-of-order execution, write buffers (writes are queued before reaching main memory), per-core cache hierarchies with delayed propagation

Different CPU architectures (x86, ARM, POWER) provide different reordering guarantees. x86 is relatively strong; ARM allows more reorderings. Language memory models must assume the weakest guarantees for portability.

📜

Java Memory Model (JMM)

ELI5

The JMM is the contract between you and the Java runtime. It says: "Follow these rules, and we guarantee these visibility properties." Without the contract, the runtime is free to do anything for performance — and your code breaks.

The JMM specifies exactly which memory writes by one thread are guaranteed to be visible to reads by another. It defines three key orderings:

  • Program Order (PO): Within a single thread, instructions appear to execute in written order
  • Synchronizes-With (SW): Pairing between sync actions — e.g., a lock release pairs with the next acquisition of the same lock
  • Happens-Before (HB): Transitive closure of PO + SW — the key guarantee for visibility

The JMM was initially weak (Java 1995) and was substantially clarified in Java 5 (2004).

⏱️

Happens-Before

ELI5

If event A happens-before event B, then everything A did is guaranteed visible to B — no stale caches, no reordering surprises. Without a happens-before relationship, there are no visibility guarantees at all.

Happens-Before (HB) is the JMM's central guarantee: if A →HB B, then the effects of A are guaranteed visible to B.

What establishes happens-before?

  • Program order: Earlier statements in the same thread HB later ones
  • Monitor unlock → next lock: Releasing synchronized HB the subsequent acquisition on the same object
  • volatile write → next read: Writing a volatile variable HB any subsequent read of it
  • Thread start/join: t.start() HB any action in thread t; any action in t HB t.join() returning
  • Transitivity: If A →HB B and B →HB C, then A →HB C
📡

volatile

ELI5

volatile is a megaphone: every write is immediately broadcast to all threads, and every read fetches the latest value from memory. But it's not a lock — it doesn't make compound operations atomic.

volatile provides visibility without mutual exclusion:

  • Writes are immediately visible to all threads (bypasses caches)
  • Reads always fetch from main memory, not a stale cache
  • Establishes happens-before: volatile write →HB subsequent volatile read of the same variable
  • Prevents compiler/hardware reordering around the access

What volatile does NOT do: it doesn't make compound operations atomic. count++ is still a race condition even on a volatile field — use AtomicInteger instead.

// Classic use: stop flag (volatile = safe, boolean = not safe)
volatile boolean stop = false;
// Thread 1:
while (!stop) { /* work */ }   // always reads fresh value
// Thread 2:
stop = true;                   // immediately visible to Thread 1

// Still a race condition even with volatile:
volatile int count = 0;
count++;  // NOT atomic — use AtomicInteger.incrementAndGet()
💥

Data Races vs Bad Interleavings

ELI5

A data race is a low-level hardware crime — two threads touching the same memory without synchronization. A bad interleaving is a logic crime — the right operations in the wrong order. Data races can cause completely nonsensical results; bad interleavings produce wrong but at least "real" values.

These are distinct problems:

  • Data Race (low-level): Concurrent read/write or write/write on the same memory location without synchronization. The JMM says reads can see completely unexpected values — including values from the future, or values that never existed. Compilers and hardware can optimize in ways that make no sense for a race-condition program.
  • Bad Interleaving (high-level): All accesses are synchronized, but the order of high-level operations is wrong. Individual reads and writes are consistent, but the program logic is violated (e.g., TOCTOU: check-then-act with a window in between).

The fix for data races: always use synchronized, volatile, or atomics on shared mutable data. The fix for bad interleavings: expand the critical section to cover the whole logical operation.

  • A data race is undefined behavior in the JMM — the compiler and CPU are free to optimize assuming it doesn't happen. The results can be completely nonsensical, not just "a bit wrong".
  • volatile is for single-variable visibility. For compound invariants (e.g., balance + transaction log must stay consistent), use synchronized.
  • Even reading a long or double is not guaranteed atomic on 32-bit JVMs — declare them volatile or use synchronized.
  • The classic broken pattern: while (!stop) {} where stop is not volatile. The compiler may hoist the read into a register and loop forever.
  • Happens-before is transitive — you can chain guarantees. synchronized gives you both happens-before and mutual exclusion.

Lock Implementations

How mutual exclusion is actually built

🤝

Peterson's Lock

ELI5

Two people want to enter a room. Each raises their hand to signal interest. But each also says "no, you go first" — writing themselves as the "victim." If you raised your hand AND you're the victim, you wait. If the other person raised their hand but they're the victim, you go in. Elegant: only one can be the victim at a time.

Peterson's Lock (1981) is the classic 2-thread software mutex, using only shared registers:

  • flag[me] = true — "I want to enter"
  • victim = me — "I yield priority to the other"
  • Wait while: flag[other] && victim == me

It proves 3 properties: mutual exclusion (at most one in CS), deadlock-freedom (at least one always progresses), and starvation-freedom (every waiting thread eventually enters).

Important: requires volatile (or equivalent) on the flag array in Java for correct memory visibility.

void lock(int me) {
    int other = 1 - me;
    flag[me] = true;             // I want in
    victim = me;                   // I yield
    while (flag[other] && victim == me) {} // spin
}
void unlock(int me) {
    flag[me] = false;            // I'm done
}
🔽

Filter Lock

ELI5

Peterson's Lock works for 2 threads. For N threads, imagine a queue of waiting rooms — levels 1 through N-1. To enter the critical section, a thread must progress through all levels. At each level, at most one thread gets "stuck" as the victim. Everyone else passes through. Eventually only one thread reaches the end.

Filter Lock generalizes Peterson's Lock to N threads. It has N-1 "levels":

  • Each level has a victim[level] variable
  • To pass level L, write yourself as victim, then wait until either no other thread is at level ≥ L, or you're no longer the victim
  • At each level, at most one thread is "held back" — the victim
  • A thread passes a level only if it's not the last to arrive

Guarantees mutual exclusion and starvation-freedom. Space cost: O(N) — one victim variable per level.

🥐

Lamport's Bakery Algorithm

ELI5

Walk into a bakery and take a number ticket. Everyone waits until their number is called — lowest number first. If two people grab the same number (a tie), the one with the lower customer ID goes first. Simple, fair, and provably correct.

Lamport's Bakery (1974) is elegant and provably correct even with regular registers (weaker than atomic):

  1. Set choosing[me] = true (entering the "bakery")
  2. Take a ticket number: number[me] = 1 + max(all numbers)
  3. Set choosing[me] = false
  4. For each other thread j: wait if j is choosing; then wait if j's ticket is lower (or equal ticket and lower ID)

Ties broken by thread ID using lexicographic ordering: (ticket_j, j) < (ticket_me, me). Starvation-free by FIFO ordering of ticket numbers.

🔑

TAS & TATAS Spinlocks

ELI5

TAS (Test-And-Set): repeatedly hammer the door with expensive atomic ops. TATAS: first peek through the window (cheap local read). Only when you see the door is free, try the expensive atomic knock. Dramatically less noise for everyone else.

Test-And-Set (TAS): A hardware instruction that atomically writes 1 to a memory location and returns the old value. A TAS spinlock spins calling TAS until it gets 0 back (was unlocked). Problem: every spin attempt is an expensive bus-locking atomic op, generating huge cache coherence traffic — performance collapses with many threads.

Test-and-Test-and-Set (TATAS): First spin reading the lock state locally (cache hit, cheap). Only attempt the atomic TAS when the local read shows the lock is free. Much less bus traffic.

TATAS + Exponential Backoff: After each failed acquisition, wait a random exponentially-growing duration before retrying. Prevents the "thundering herd" — all threads detecting a free lock and colliding simultaneously.

🔄

Compare-And-Swap (CAS)

ELI5

CAS says: "Only update this value if it still equals what I expected." If the value changed since I last read it, the operation fails and I retry. It's the hardware primitive behind most lock-free and atomic data structures.

CAS atomically: reads the current value, compares it to an expected value, and only writes the new value if they match. Returns whether the swap succeeded.

// Pseudocode — executed atomically by hardware
bool CAS(addr, expected, newVal) {
    if (*addr == expected) { *addr = newVal; return true; }
    return false;
}
  • Foundation of AtomicInteger, AtomicReference, ConcurrentHashMap
  • Used in lock-free algorithms — threads retry on failure instead of blocking
  • On x86: CMPXCHG instruction. On ARM: LL/SC (load-linked/store-conditional) pair
  • ABA problem: CAS can't detect if a value changed from A→B→A. Use stamped references when this matters.
  • Peterson's and Bakery algorithms require volatile (or AtomicIntegerArray) in Java — plain arrays won't provide the visibility guarantees the algorithms require.
  • Software locks (Peterson, Filter, Bakery) are O(N) space. Hardware CAS/TAS locks are O(1). In practice, always use hardware-backed locks.
  • Spinlocks are only efficient when the critical section is very short and contention is low. For long waits, blocking locks (OS-mediated) are better — the CPU can do useful work.
  • Backoff is crucial for TATAS under high contention. Without it, all threads detect a free lock simultaneously and collide — the "thundering herd".
  • CAS-based lock-free algorithms avoid deadlock by design — a thread that fails CAS simply retries. But they can still livelock or starve under extreme contention.

Semaphores, Barriers & Virtual Threads

Higher-level coordination primitives

🔢

Semaphores

ELI5

A semaphore is a counter with two operations: acquire (take a ticket, wait if none are left) and release (add a ticket back). A semaphore initialized to 1 is a mutex — a semaphore initialized to N controls access to N identical resources.

A semaphore is a non-negative integer counter with two atomic operations:

  • acquire(S): If S > 0, decrement and proceed. If S = 0, block until it increases.
  • release(S): Increment S, unblocking one waiting thread if any.

Use cases:

  • Binary semaphore (S=1): Equivalent to a mutex lock
  • Counting semaphore (S=N): Allows up to N threads to access a resource simultaneously (e.g., connection pool of size N)
  • Signaling: One thread releases to signal an event; another acquires to wait for it

Java: java.util.concurrent.Semaphore

🤝

Rendezvous Pattern

ELI5

Two friends agree to meet before going into the restaurant. Neither goes in until both have arrived. Each waits for the other's signal — but there's a subtle trap: if you wait before signaling, you both wait forever.

Rendezvous ensures two threads reach a synchronization point before either continues. It needs two semaphores (one per thread), both initialized to 0:

  1. Thread A: release(arrived_A) then acquire(arrived_B)
  2. Thread B: release(arrived_B) then acquire(arrived_A)

Classic mistake: if both threads acquire first, then try to release — they deadlock, each waiting for the other to signal first.

Key rule: always signal (release) your own arrival before waiting (acquire) for the other's — this breaks the circular wait.

🛑

Barriers

ELI5

A barrier is a finish line for a team race: nobody crosses until everyone finishes their lap. The last runner to arrive "opens the gate" and all runners proceed together to the next lap.

A barrier forces all N threads to wait at a checkpoint until every thread has arrived. The last to arrive releases all waiters.

Building a correct barrier is surprisingly tricky — there are several failure modes:

  • Race on counter: Incrementing a shared count with count++ is not atomic → use a mutex
  • Only one thread released: A single release() only frees one waiter → use notifyAll() or N-1 releases
  • Non-reusability: Fast threads in the next iteration can race past a barrier before slow threads have finished resetting it → requires a two-phase design

Two-phase reusable barrier: Phase 1 (all arrive), Phase 2 (all reset state). Only when all threads have completed phase 2 can phase 1 restart. Java: CyclicBarrier implements this correctly.

// Java's built-in CyclicBarrier — reusable, correct
CyclicBarrier barrier = new CyclicBarrier(N);

// Each thread, at the synchronization point:
barrier.await(); // blocks until all N threads call await()
                 // then all N proceed simultaneously
🗺️

Deadlock Handling Strategies

ELI5

Three ways to deal with deadlock: prevention (design the system so circular wait can never happen), detection + recovery (let it happen, then find and break cycles), or avoidance (dynamically check before each allocation if it's safe). Prevention is almost always the right choice.

Resource Ordering (Prevention): Define a global total order on all locks. Every thread must acquire locks in ascending order. This structurally breaks the circular-wait Coffman condition.

// Always lock the lower-numbered account first
if (this.id < other.id) {
    synchronized(this) { synchronized(other) { transfer(); } }
} else {
    synchronized(other) { synchronized(this) { transfer(); } }
}

Two-Phase Locking (Avoidance): Try to acquire all needed locks. If any fails, release all and retry. Works but can livelock under heavy contention.

Detection + Recovery: Build a resource allocation graph; scan for cycles periodically. If found, abort a thread to break the cycle. Used in databases; impractical for general parallel code (aborting a thread leaves inconsistent state).

🪄

Virtual Threads (Project Loom)

ELI5

Normal threads are like renting entire hotel rooms — expensive, limited supply. Virtual threads are like hot-desking: millions of "workers" share a small pool of physical desks. When a virtual thread blocks (waiting for I/O), it vacates the desk so another can use it. No blocking a physical OS thread while waiting.

Virtual threads (Java 21, Project Loom) decouple application concurrency from OS thread limits:

  • Platform threads: 1:1 with OS threads, ~1MB stack each, max thousands
  • Virtual threads: JVM-managed, heap-allocated state, max millions
  • Carrier threads: The real OS threads. Virtual threads mount/unmount onto carriers. On I/O block → unmount, carrier serves another virtual thread.

Key rules:

  • Never pool virtual threads — they're cheap, create per task instead
  • Ideal for I/O-bound workloads (servers, DB queries, network calls)
  • Minimal benefit for CPU-bound work — they occupy a carrier regardless
  • Create with: Thread.ofVirtual().start(task) or Executors.newVirtualThreadPerTaskExecutor()
  • In the rendezvous pattern, always release before acquire — signaling your arrival before waiting for the other's. The reverse deadlocks.
  • Prefer CyclicBarrier over rolling your own barrier — correctness requires a two-phase design that's easy to get wrong.
  • For lock ordering: if objects don't have a natural order, use System.identityHashCode() or an AtomicLong counter to assign unique IDs at construction time.
  • Virtual threads are not faster for CPU-bound work. Their benefit is scaling I/O-bound concurrency without wasting OS threads on blocking.
  • Virtual threads use ForkJoinPool internally as the carrier pool — don't block virtual threads with synchronized on long I/O (use ReentrantLock instead, which allows unmounting).

Monitors

Producer-Consumer, Bounded Buffers & the Monitor Pattern

🏭

Producer-Consumer Pattern

ELI5

Imagine a bakery conveyor belt: bakers (producers) slide bread onto the belt; customers (consumers) take it from the other end. Neither side coordinates directly — the belt is the shared buffer.

A fundamental concurrency pattern: producer threads generate items and put them in a shared buffer; consumer threads pull items out and process them.

  • Decouples producers and consumers so they can run at different speeds
  • Enables pipeline architectures — one stage's output is the next stage's input
  • Core challenge: block producers when the buffer is full, block consumers when it's empty
  • Web server: network I/O threads produce requests → worker threads consume & handle them
  • Video player: decoder produces frames → renderer consumes & displays them
  • Logger: app threads produce log lines → background writer consumes & flushes to disk
🔁

Bounded Buffer

ELI5

A circular conveyor belt at a sushi restaurant with a fixed number of plate spots. Chefs stop adding when it's full; customers wait when it's empty. Plates cycle around without ever needing a "reset".

Implemented as a circular array with two index pointers:

  • in: next write position — advances with (in+1) % capacity
  • out: next read position — advances the same way
  • One slot stays empty to tell "full" apart from "empty" — full when (in+1)%cap == out
🔢

Semaphore-Based Solution

ELI5

Three counters on a whiteboard: "items ready", "empty slots left", and "someone's using it now". Check the right counter before touching the shelf — never grab the key while still waiting for stock.

Three semaphores manage access correctly:

  • nonEmpty: counts available items — consumers acquire() before taking
  • nonFull: counts empty slots — producers acquire() before adding
  • mutex: binary semaphore for exclusive buffer access

Golden rule: always acquire the condition semaphore before the mutex. Holding the mutex while blocking on a condition prevents anyone else from signaling it — deadlock.

🏛️

What is a Monitor?

ELI5

A bank teller booth: only one customer at a time (mutual exclusion), and if you need something that isn't ready yet, you step aside and wait on a bench until the teller calls you back.

A monitor is a high-level abstraction that bundles data, mutual exclusion, and condition waiting into one object:

  • Encapsulation: only monitor methods can access shared data
  • Implicit mutual exclusion: only one thread runs inside at a time
  • Condition variables: threads can wait for state changes without polling

In Java, any synchronized object is a monitor. Cleaner than scattered lock/unlock calls.

☕

Java wait / notify / notifyAll

ELI5

wait() = take a number and sit down. notify() = call one random number. notifyAll() = call all numbers at once. Everyone woken rushes back, but only one gets served at a time.

Java's built-in monitor primitives (must be called inside synchronized):

  • wait(): atomically releases lock + suspends thread until notified
  • notify(): wakes one arbitrary waiting thread
  • notifyAll(): wakes all — safer default choice

Critical: use while, not if. After waking, another thread may have stolen the condition before you reacquire the lock. The while loop re-checks and waits again if needed.

  • synchronized void enqueue(T x) { while (isFull()) wait(); buf[in]=x; in=(in+1)%cap; notifyAll(); }
  • synchronized T dequeue() { while (isEmpty()) wait(); T v=buf[out]; out=(out+1)%cap; notifyAll(); return v; }
🎛️

Condition Objects

ELI5

Instead of one waiting room for everyone, you get separate waiting rooms — producers wait in "buffer full", consumers wait in "buffer empty". Now you can wake exactly the right group.

java.util.concurrent.locks.Condition provides multiple wait sets on one lock:

  • Create: Condition notFull = lock.newCondition();
  • await() = wait(), signal() = notify(), signalAll() = notifyAll()
  • Producers signal() only the notEmpty condition; consumers signal only notFull — no wasted wakeups
  • Always wrap in try-finally to guarantee lock.unlock()
  • Always use while, never if, around wait()/await() — spurious wakeups and "signal and continue" semantics mean the condition may be false again when you resume.
  • Default to notifyAll() / signalAll(). notify() can wake the "wrong" thread, leaving the correct waiter stranded forever.
  • Separate Condition objects (one per role) let you use targeted signal() safely and eliminate unnecessary wakeups.
  • Never spin-wait while holding a monitor lock — the lock prevents anyone else from changing the condition you're waiting on. Use wait() to release it.
  • In semaphore solutions: always acquire the condition semaphore before the mutex. Reversing the order causes deadlock.

R/W Locks & Lock Granularity

Sleeping Barber, Reader-Writer Locks & Fine-Grained Locking

🔑

Explicit Locks & Conditions

ELI5

Instead of a door that auto-locks (synchronized), you get a manual deadbolt (ReentrantLock) — more control, but you must remember to unlock it no matter what happens.

ReentrantLock + Condition gives more control than intrinsic monitors:

  • Always unlock in a try-finally block — exceptions must not leave the lock held
  • Use while loops around await() to guard against spurious wakeups
  • Call signal() on the specific condition queue (not signalAll) for efficiency when roles are distinct
💈

Sleeping Barber & Lost Wakeups

ELI5

A barber sleeps when idle. A customer arrives, sees the barber sleeping, and shouts "Wake up!" — but if the barber hasn't dozed off yet, the shout is wasted. When the barber finally naps, nobody wakes him. Signal lost!

The lost wakeup problem: a signal sent before the receiver calls await() is silently discarded.

  • Fix: track waiting threads with counters (n for items, m for empty slots)
  • Only call signal() when the counter indicates threads are actually blocked
  • This avoids wasting signals and prevents the wakeup-before-sleep race
  • Wrong: signal() is called before the thread runs await() → wakeup is lost → thread sleeps forever
  • Fix: increment a counter before signaling; the waiting side decrements before sleeping and checks the counter on wake
📚

Reader/Writer Locks

ELI5

A library book: many people can read it simultaneously, but if someone wants to write notes in it, everyone else must step back until they're done. Reading is shared; writing is exclusive.

Optimized for workloads where reads vastly outnumber writes:

  • Read lock: multiple threads can hold simultaneously — great for hashtable lookups
  • Write lock: exclusive — no readers or other writers allowed
  • Invariant: writers == 0 || readers == 0 with writers ≤ 1
  • Fairness tradeoff: readers can starve writers in naive implementations; writer-priority risks starving readers
  • Java: ReentrantReadWriteLock — note: does not support read-to-write upgrade
🔭

Lock Granularity Spectrum

ELI5

One master key for the whole building (coarse) is simple but creates a queue. Individual keys per room (fine) is faster but you need more keys and can accidentally deadlock by grabbing them in the wrong order.

Five levels on the spectrum from simple → complex:

  • Coarse-grained: one lock for the whole structure — simple, low concurrency
  • Fine-grained: one lock per component — higher concurrency, higher complexity
  • Optimistic: traverse unlocked, validate before committing — avoids most lock acquisitions
  • Lazy: mark deletions logically first, clean up later — simplifies remove logic
  • Lock-free: atomics only, no locks — highest concurrency, hardest to implement
🤲

Hand-Over-Hand Locking

ELI5

Climbing a rope: you grip the next knot before releasing the one below. You're always holding at least one knot — safe to move forward, and others can climb the lower part simultaneously.

Safe traversal of linked lists under fine-grained locking:

  • Always hold the current node's lock while examining it
  • When moving forward: lock the next node, then release the current one
  • Never release current before locking next — a concurrent delete could corrupt the traversal
  • Modifications across multiple nodes require holding all affected locks simultaneously
  • Always unlock in a try-finally block. An exception without finally can leave the lock permanently held.
  • ReentrantReadWriteLock does not support read-to-write upgrades — attempting one deadlocks. Upgrade means: release read lock, acquire write lock (but now the state may have changed!).
  • For R/W fairness, use new ReentrantReadWriteLock(true) (fair mode) to prevent writer starvation — at the cost of some throughput.
  • In hand-over-hand locking, always lock in the same direction (head → tail) to avoid deadlock between concurrent traversals.
  • Prefer library implementations (java.util.concurrent) over custom synchronization in production — they handle edge cases you will miss.
🎯
Module 13

Optimistic & Lazy Sync

Skip Lists, Lock-Free Motivation & Progress Guarantees

🔬

Fine-Grained Locking

ELI5

Instead of one lock on the whole list, each node has its own tiny lock. Multiple threads can work on different parts of the list simultaneously — but you have to lock two nodes at once to safely connect or remove one.

Split the data structure so each piece has its own lock — increases concurrency but adds overhead:

  • Traversal requires acquiring and releasing many locks (hand-over-hand)
  • A slow thread holding one node's lock blocks threads needing that node — even if they want a different operation
  • Correct, but the constant lock churn limits scalability
🤞

Optimistic Synchronization

ELI5

Walk the store without a cart (no lock), grab what you want, then check at the checkout whether anyone moved things while you were browsing. If the shelves changed, go back and try again.

Assume conflicts are rare — traverse without locks, then verify before committing:

  • Step 1: traverse lock-free to find target nodes
  • Step 2: lock only the target nodes
  • Step 3: validate — is predecessor still reachable? Is it still the immediate neighbor? Are nodes unmarked?
  • Step 4: if validation passes, perform operation; otherwise unlock and retry

Lock-free traversal allows concurrent readers, but may require double traversal. Not starvation-free — threads can retry indefinitely under high contention.

🏷️

Lazy Synchronization

ELI5

When removing a book from a library catalogue, you first put a sticky note saying "removed" (logical delete). The actual physical removal from the shelf happens later. Anyone can see the note immediately without needing to wait.

Separate logical deletion (marking) from physical deletion (pointer surgery):

  • Remove: scan, lock both nodes, validate, mark node as deleted, update predecessor's pointer, unlock
  • Key invariant: every unmarked node is reachable from head via unmarked nodes
  • contains(): wait-free — single scan, no locks, just checks the mark flag
  • Marked nodes linger in memory until physical removal, but contains() skips them
🪜

Lazy Skip Lists

ELI5

A multi-deck highway: the top deck skips large distances, lower decks fill in the details. You drop down a level whenever you overshoot. No rebalancing needed — each node's height is randomly chosen at insertion.

Probabilistic multi-level linked list — chosen over balanced trees for easier concurrent implementation:

  • Each node has random height: P(height=n) = 0.5ⁿ — expected O(log n) search
  • contains(): standard traversal, no locks, checks unmarked + fully-linked — fully wait-free
  • add(): find predecessors lock-free → lock them → validate → splice node at all levels → mark "fully linked" → unlock
  • remove(): find lock-free → lock victim → mark deleted → lock predecessors → update pointers → unlock
  • No rebalancing ever needed — probabilistic balance is good enough
⚡

Why Locking Hurts

ELI5

Locks are like traffic lights: useful, but red lights waste time even when no other cars exist. Under heavy traffic they cause jams. And if a driver crashes at the intersection while holding green, everyone is stuck forever.

Fundamental problems with all locking approaches:

  • Pessimistic: mutual exclusion is enforced even when there's no real conflict
  • Priority inversion: a high-priority thread blocked on a lock held by a low-priority thread
  • Convoying: one preempted lock holder delays every waiting thread
  • Deadlock & livelock: possible whenever multiple locks are involved
  • Fault intolerance: a thread dying while holding a lock leaves it permanently held
🏁

Lock-Free vs Wait-Free

ELI5

Lock-free = at least one runner always finishes each lap, but some might run forever. Wait-free = every runner is guaranteed to finish in bounded laps — nobody starves.

Two levels of non-blocking progress guarantees:

  • Lock-free: system-wide progress — at least one thread makes progress in finite steps. Individual threads may starve.
  • Wait-free: per-thread progress — every thread completes in finite steps regardless of others. Strongest guarantee, hardest to implement.

  • Optimistic sync trades double traversal for fewer lock acquisitions — only wins when conflicts are rare and traversal is cheap.
  • Lazy sync's key insight: making contains() wait-free is a massive win when reads dominate (e.g., concurrent sets, caches).
  • Skip lists are preferred over concurrent balanced trees in practice because they avoid global rebalancing — which is extremely hard to do lock-free.
  • Optimistic sync is not starvation-free — under constant contention, a thread may retry forever. Add a backoff or cap retries.
  • "Lock-free" ≠ "faster". Lock-free structures often have higher constant factors. Their advantage is progress guarantees under adversarial scheduling, not raw speed.

Module 14

Lock-Free Structures

Stack, List, Queue — Built with CAS and Atomic References

🧲

Atomic References & CAS

ELI5

CAS is like a conditional sticky note swap: "Replace this note with a new one, but only if it still says what I last read." If someone else changed it in the meantime, your swap fails and you try again.

Compare-and-Swap (CAS): atomically reads a location, compares to expected value, writes new value only if they match. Returns success/failure.

  • AtomicReference<T>: wraps an object reference for safe CAS operations on pointers
  • AtomicMarkableReference<T>: combines a reference + boolean flag in one atomic unit — enables logical deletion without a separate lock
  • On failure: read the new value and retry (compare-and-swap loop)
  • Lock-free algorithms are built entirely on these primitives
📚

Lock-Free Stack

ELI5

To push: create a new sticky note pointing to the current top, then try to atomically swap the top pointer. If someone else pushed first, your swap fails — reread the top and try again.

Uses an AtomicReference<Node> for the top pointer:

  • push: create node, set node.next = top.get(), then top.compareAndSet(node.next, node) — retry on failure
  • pop: read head = top.get(), then top.compareAndSet(head, head.next) — retry on failure
  • Deadlock-free by construction — no locks exist to deadlock on
  • Under contention, many threads retry and waste cycles → use exponential backoff
🔗

Lock-Free List (Lazy Deletion)

ELI5

To delete a node: first mark it (put a red dot on it) without touching any pointers, then swing the predecessor's pointer past it. Any thread that sees the red dot knows the node is gone and helps clean it up.

AtomicMarkableReference combines next-pointer + deleted-flag atomically:

  • Logical delete: CAS the node's own next reference to set the mark bit
  • Physical delete: CAS the predecessor's next to skip the marked node
  • Helping: threads encountering a marked node attempt physical deletion before proceeding — ensures lock-free progress
  • Add: traverse, CAS predecessor's next to new node — retry if predecessor is marked
🚌

Lock-Free Queue

ELI5

A bus stop queue: to join, you atomically link yourself to the last person. To leave, you atomically advance the front pointer. A dummy person always stands at the front so the queue is never truly empty.

Michael & Scott queue — uses a sentinel node (dummy head) and two atomic pointers:

  • Enqueue: CAS the tail's next to the new node, then advance the tail pointer
  • Dequeue: read head's next (actual first item), CAS head to that node, return its value
  • Empty when head.next == null
  • The sentinel prevents the head/tail confusion when the queue has exactly one item
⏳

Exponential Backoff

ELI5

After a failed CAS, don't immediately try again — you'll just collide with the same threads. Wait a random short time, then try again. Double the wait after each failure. Eventually, everyone spreads out and stops colliding.

Without backoff, contended lock-free structures can underperform locks due to cache-line thrashing:

  • On CAS failure: sleep for a random interval in range [0, cap]
  • Double cap after each failure up to a maximum
  • Dramatically improves throughput under contention by spacing out retries
  • Does not affect correctness — only performance
  • Lock-free ≠ contention-free. Under heavy contention, failed CAS attempts generate cache traffic. Exponential backoff is almost always needed for production performance.
  • AtomicMarkableReference is the key to lock-free list deletion — it atomically packages "where to go next" and "am I deleted" so neither can be changed independently.
  • The helping mechanism is what makes lock-free structures truly lock-free: if a thread is preempted mid-operation, other threads detect the partial state and complete the operation on its behalf.
  • Beware the ABA problem (next module) — it can silently corrupt lock-free structures when memory is reused.
  • Java's garbage collector implicitly solves ABA for object references — pointer recycling never returns the same object identity.
🔗
Module 15

Lock-Free Queue Protocol

Michael & Scott Algorithm — Deep Dive

👻

Sentinel Node Strategy

ELI5

The queue always has a permanent placeholder at the front that holds no real data. The queue is empty when this placeholder's "next" pointer is null. It eliminates all the special-case headaches of a truly empty queue.

A dummy node always lives at the head, never removed:

  • Empty condition: sentinel.next == null — clean and unambiguous
  • Head always points to the sentinel; the sentinel's next is the actual first item
  • When dequeuing: the sentinel's next node becomes the new sentinel — it takes the role, not the position
  • Eliminates corner cases when head and tail point to the same node
📥

M&S Enqueue Protocol

ELI5

Joining a queue: you tap the last person's shoulder (link yourself as their next). Then you try to tell the "tail" sign to point to you. If someone else joined first, help them update the sign, then try again.

Three steps, each guarded by CAS:

  • Step 1: read tail and tail.next
  • Step 2: if tail.next != null — tail is stale (someone half-enqueued). Advance it: CAS(tail, tail.next) and retry
  • Step 3: CAS(tail.next, null, newNode) — link new node. Then CAS(tail, tail, newNode) — advance tail

The "advance stale tail" step is the helping mechanism — even if the original thread died, progress continues.

📤

M&S Dequeue Protocol

ELI5

Leaving a queue: check if you're the only one there (empty). If not, try to atomically advance the "head" sign to the next person. If the sign moved while you were looking, try again.

Four steps:

  • Step 1: read head, tail, and head.next
  • Step 2: if head == tail and head.next == null → truly empty, return empty
  • Step 3: if head == tail and head.next != null → tail is stale, advance it first
  • Step 4: read value from head.next, then CAS(head, head, head.next) — advance head
🤝

Helping & Transient Inconsistency

ELI5

The queue can be in a half-finished state — like a relay race where the baton is mid-air. Any runner who sees this picks up the baton and completes the handoff before doing their own run. Nobody waits.

The M&S algorithm is lock-free because of the helping pattern:

  • Transient inconsistency: between linking a new node and advancing the tail pointer, the queue is "in progress" — not illegal, just incomplete
  • Helping: any thread detecting a stale tail advances it before proceeding. No thread needs to wait for the original thread to finish.
  • Contrast with locks: a thread that dies mid-operation with a lock held blocks everyone forever. Here, others complete the job.
  • This is why the algorithm is truly lock-free — progress doesn't depend on any individual thread
  • The sentinel node is the single most important simplification in lock-free queues. Never try to design without one.
  • Always re-read tail after finding tail.next != null — advancing a stale tail is a courtesy to others, but your own enqueue still needs a fresh snapshot.
  • The M&S queue has a subtle quirk: tail can lag behind the real last node by one step. This is intentional and safe — the helping mechanism covers it.
  • In non-GC languages (C/C++), the M&S algorithm requires hazard pointers or pointer tagging to prevent ABA corruption on node reuse.

Module 16

ABA & Linearizability

Memory Hazards in Lock-Free Code & Formal Correctness

👁️

The ABA Problem

ELI5

You see a $5 bill on a table, get distracted, and come back — it's still a $5 bill. But unknown to you, it was picked up and a different $5 bill was placed there. Your CAS succeeds thinking nothing changed, but it's a different bill.

CAS only checks the value, not the identity or history of a location:

  • Thread X reads head = A, gets preempted
  • Thread Y pops A, pushes B, then pushes a recycled node at address A
  • Thread X resumes: CAS(A, B) succeeds — but A now refers to different data
  • Result: silent data corruption — B is lost, the stack is broken
  • Only matters when memory is recycled. Java's GC prevents it for object references.
🏷️

DCAS & Pointer Tagging

ELI5

Add a serial number to the bill. Even if the same $5 denomination returns to the table, its serial number is different. Now CAS checks both the value and the serial number — a recycled node fails the check.

Two solutions that add a version/tag alongside the pointer:

  • DCAS (Double CAS): atomically updates a pointer + a version counter. Version increments on every write. Java: AtomicStampedReference (integer stamp) or AtomicMarkableReference (boolean)
  • Pointer Tagging: embeds version bits in the unused lower bits of aligned pointers (e.g., bottom 3 bits are always 0 for 8-byte-aligned data). CAS checks both address and tag bits simultaneously.

Both make it astronomically unlikely for a recycled address to match the old stamp.

⚠️

Hazard Pointers

ELI5

Before touching a node, you write its address on a public whiteboard. Before the GC recycles any node, it checks every whiteboard — if a node is listed by anyone, it cannot be freed yet.

A manual reference-counting scheme for C/C++ lock-free code:

  • Each thread has a small array of hazard pointers — public declarations of "I'm using this node"
  • Set hazard[me] = ptr before dereferencing. Then re-verify the pointer is still valid after setting — it may have changed in the gap.
  • Before freeing a node, scan all hazard arrays — if any thread declared it hazardous, defer the free
  • Clear your hazard pointer when done
  • Complex, but essential in non-GC environments. In Java, GC overhead usually exceeds the benefit.
📍

Linearizability

ELI5

A concurrent object is linearizable if it looks as if every operation happened atomically at one instant, even though operations actually overlap in time. Like a court that reconstructs events as if each happened at a single precise moment.

The standard correctness condition for concurrent objects:

  • Each operation must appear to take effect at a single linearization point between its invocation and its response
  • The resulting order of operations must be a valid sequential history of the object
  • Real-time preservation: if operation A completes before B starts, A must precede B in the sequential history
  • Linearizability is composable — if each component is linearizable, the system is too
🔎

Linearizability Examples

ELI5

To check if a history is linearizable: try to find one instant per operation where they could have "actually happened" — all in order, all making sense. If no such assignment exists, the history is broken.

  • Linearizable queue: enq(x), enq(y), deq()→x, deq()→y — respects FIFO; x was enqueued before y and dequeued first ✓
  • Non-linearizable: enq(y), deq()→y where deq response comes before enq(y) invocation — dequeued something not yet enqueued ✗
  • Linearization points are sometimes fixed (CAS at a specific line) and sometimes execution-dependent (dequeue from an empty vs non-empty queue)
  • In Java, ABA on object references is impossible — the GC never reuses an object's identity. You only need stamps/tagging for raw pointer types (C/C++).
  • AtomicStampedReference fixes ABA but is slow — it boxes both reference and stamp into a heap object. Use it only when ABA is a real threat.
  • When proving linearizability, first identify each operation's linearization point. For CAS-based algorithms, it's usually the successful CAS instruction.
  • Linearizability is the strongest standard correctness condition for concurrent objects. If you can prove it, you can compose your component safely with others.
  • GC is the simplest ABA solution — another argument for writing concurrent data structures in garbage-collected languages whenever performance allows.
⚖️

Consistency Models

Linearizability Formalism, Sequential Consistency & Quiescent Consistency

📜

Linearizability — Formal Definition

ELI5

Imagine a spy who watches all your concurrent operations and must write a plausible one-by-one report of what happened. If she can write a report that is consistent with both the rules of the object and with who-finished-before-who, the system is linearizable.

Formally: a history H is linearizable iff it can be extended to G such that:

  • complete(G) is equivalent to some legal sequential history S
  • Real-time preservation: if operation m₀ completes before m₁ starts in H, then m₀ precedes m₁ in S
  • A history is a sequence of invocation and response events from all threads
  • A linearization point is the single instant within each operation where it takes effect
  • Linearization points are sometimes fixed (always at the CAS), sometimes execution-dependent
📋

Sequential Consistency

ELI5

Operations must appear in some overall order that respects each thread's own program order — but operations from different threads can be freely interleaved in time. No wall-clock requirement.

A weaker model than linearizability — drops the real-time constraint:

  • Operations can be reordered across threads as long as each thread's own order is preserved
  • An operation that finished before another started can appear after it in the sequential history
  • Critical limitation: SC is not composable — individually SC components don't guarantee a system-wide SC execution
  • Modern CPUs guarantee at most SC for their memory model — some guarantee even less
😴

Quiescent Consistency

ELI5

Operations must appear in some order — but only operations that happen before a complete quiet period must precede operations after it. Overlapping operations can appear in any order.

The weakest of the three models — only orders across quiescent states (moments with no pending operations):

  • Operations separated by a quiescent state must appear in real-time order
  • Concurrent (overlapping) operations can appear in any order
  • Composable (like linearizability) — but much weaker in its ordering guarantees
  • Practical for high-throughput counters and queues where strict ordering isn't needed
🏆

Model Comparison

ELI5

Three referees watching the same game: Linearizability writes down every play with exact timestamps. Sequential Consistency writes them in a plausible story order, ignoring exact times. Quiescent Consistency only cares about the order between halftime breaks.

Cost Higher Moderate Lower

Modern CPUs guarantee memory models weaker than SC — explicit volatile, fences, or atomics are needed to strengthen guarantees.

  • Linearizability is the only composable strong model. If you need to combine multiple concurrent objects safely, each one must be linearizable.
  • Sequential Consistency's non-composability is a practical showstopper — two SC objects together can produce a non-SC execution. This is why it's rarely used as a formal spec.
  • When verifying linearizability, draw a timeline. Mark each operation as a bar from invocation to response. Try to place one point per bar where the "effect" happens, consistent with the object's sequential semantics.
  • Quiescent Consistency is the right model for benchmarks and performance counters — you only care about values at sampling points (natural quiescent states), not mid-flight.

Module 18

Consensus & Transactional Memory

Primitive Power Hierarchy & Programming without Locks

🗳️

The Consensus Problem

ELI5

N friends each suggest a number. They must all agree on exactly one of the suggested numbers — no matter how slow or unpredictable the network is. The challenge: how powerful does your communication primitive need to be?

N threads each propose a value; they must agree on exactly one proposed value:

  • Consistency: all threads decide the same value
  • Validity: the decided value was proposed by some thread
  • Wait-freedom: every thread decides in finite steps regardless of others
  • Consensus is the fundamental building block for any distributed coordination protocol
🏔️

Consensus Number Hierarchy

ELI5

Each primitive has a power level — how many threads it can coordinate to consensus wait-free. Atomic registers: level 1 (barely useful). CAS: level ∞ (unlimited power). You can't build a higher level from a lower one.

The consensus number of a primitive is the max threads it can solve consensus for wait-free:

  • CN = 1: Atomic registers — only one thread can decide
  • CN = 2: Test-and-Set, Fetch&Add, FIFO Queue — two threads only
  • CN = ∞: Compare-and-Swap (CAS), LL/SC — any number of threads

Impossibility: a primitive with CN=k cannot build a wait-free implementation of an object with CN > k. You cannot build a wait-free queue from atomic registers alone.

🧾

Transactional Memory (TM)

ELI5

Like a database transaction: wrap your operations in atomic { ... }. If no conflict, changes commit all-at-once. If there's a conflict, the system automatically rolls back and retries — no deadlocks, no manual locks.

Shift synchronization burden from programmer to the runtime:

  • Atomicity: changes are visible all-at-once or not at all
  • Isolation: each transaction sees a consistent snapshot
  • Serializability: parallel execution equivalent to some serial order
  • vs Locks: no deadlocks, no priority inversion, composable
  • vs Lock-Free: no ABA problem, no manual CAS loops — far simpler to write
  • Implementations: Hardware TM (HTM, fast but limited), Software TM (STM, flexible, higher overhead)
⚙️

TM Design Choices

ELI5

STM needs to answer: what counts as a conflict? (track reads and writes), when to detect it? (eagerly or at commit), and what to do? (abort and retry). Different answers give different performance tradeoffs.

  • Conflict detection: read-set and write-set tracking; two transactions conflict if one reads what another writes
  • Nesting: flattened (inner merges with outer) vs closed (inner can abort independently)
  • Scope: all variables vs reference-based (Scala-STM wraps shared data in Ref objects)
  • I/O problem: operations with irreversible side effects (printing, network) cannot be rolled back — STM cannot safely wrap them
  • HTM (Intel TSX) is mostly disabled due to security vulnerabilities (Spectre-class side channels)
🕰️

Clock-Based STM

ELI5

Every transaction gets a birthday timestamp. Every memory location remembers when it was last changed. If you read something changed after your birthday, your snapshot is stale — abort and retry with a new birthday.

Uses a global version clock to detect stale reads:

  • Transaction gets a birthdate = current clock value at start
  • Read: check object's timestamp ≤ birthdate — if newer, abort (someone changed it since our snapshot)
  • Commit: lock all read/write objects → validate all reads → increment clock → write new values with new timestamp → unlock
  • Lock objects in consistent order to prevent deadlock during commit
  • retry() in Scala-STM / Haskell STM: abort and sleep until any watched variable changes — elegant solution to conditional waiting
  • The consensus number hierarchy proves theoretical limits: no matter how clever you are, you cannot build a wait-free queue from only read/write registers. You need at least CAS.
  • STM's biggest practical limitation is I/O inside transactions. Never print, log to disk, or send network packets inside an atomic block — there's no way to undo them on abort.
  • Scala-STM's retry() is the equivalent of monitor wait() — it aborts the transaction and re-runs it when any of its read variables change.
  • HTM (Intel TSX) was disabled on most CPUs after Spectre vulnerabilities. Don't rely on it being available in production environments.
  • STM conflict rates matter enormously for performance. Low-conflict workloads run near-optimally; high-conflict workloads (many writes to shared data) can be slower than simple locking.
🌐
Module 19

Distributed Memory & MPI

Message Passing, Go Channels & High-Performance Computing

🖧

Distributed Memory Architecture

ELI5

Instead of all workers sharing one giant whiteboard (shared memory), each worker has their own private whiteboard. To share information, they must explicitly pass notes through a network — no one can read another's board directly.

Two fundamentally different parallel architectures:

  • Shared memory: multiple CPUs access a common memory pool — easy to program, limited scalability (hundreds of cores)
  • Distributed memory: each node has private CPU + memory, connected via network — scales to millions of cores (HPC clusters), but requires explicit communication
  • Distributed memory eliminates all shared-state hazards (races, locks) — but communication latency and bandwidth become the new bottleneck
  • Also motivates functional programming (immutable data) and message passing (private state + explicit exchange)
✉️

Message Passing

ELI5

Instead of shared memory, tasks own their data privately and cooperate by sending messages. Like employees at different offices who work independently but email each other when they need to share results.

State lives inside each task; communication is explicit:

  • Synchronous send: blocks until the receiver accepts — like a hand-to-hand pass. Guarantees delivery but can deadlock if both sides wait for each other.
  • Asynchronous send: returns immediately; message queued in a mailbox. Receiver retrieves later. More flexible, but requires bounded buffers.
  • No shared mutable state means no data races, no locks — a completely different programming model
🐹

Go Concurrency Model

ELI5

Go's motto: "Don't communicate by sharing memory; share memory by communicating." Goroutines are tiny threads. Channels are typed pipes between them — pass data through the pipe instead of sharing a variable.

  • Goroutines: lightweight concurrent functions, multiplexed onto OS threads by the runtime — far cheaper than Java threads
  • Channels: typed, directional communication pipes
    • Unbuffered: synchronous — sender blocks until receiver is ready
    • Buffered: asynchronous up to buffer capacity
  • Risk: synchronous channels can deadlock if a goroutine waits on channel A while the expected receiver waits on channel B
  • select statement: wait on multiple channels simultaneously — the first ready one proceeds
🔌

Message Passing Interface (MPI)

ELI5

A standardized postal service for HPC clusters. Every process gets a unique rank (ID). They run the same code but use their rank to decide what role to play — like one program that branches based on "am I the manager or a worker?"

The standard for distributed-memory parallel computing:

  • Communicators: groups of processes (e.g., MPI_COMM_WORLD = all processes)
  • Rank: unique integer ID 0 to size−1 per process
  • SPMD model: all processes run identical code; if (rank == 0) { ... } distinguishes roles
  • Point-to-point: MPI_Send(buf, count, type, dest, tag, comm) / MPI_Recv(..., src, ...)
  • Collective: MPI_Bcast, MPI_Reduce, MPI_Scatter, MPI_Gather — operate on all processes in a communicator
⚡

Data Locality & HPC Performance

ELI5

In modern HPC, moving data costs more than computing. A multiplication is nearly free — moving a number from far RAM to the CPU core costs hundreds of multiplications. The real game is keeping data close to where it's being processed.

Modern HPC treats computation as a data management challenge:

  • Energy cost of fetching data from DRAM ≫ cost of floating-point arithmetic
  • Spatial locality: access memory addresses near each other — cache lines are 64 bytes wide
  • Temporal locality: reuse data while it's still in cache — avoid re-fetching
  • Control locality: predictable branches and tight loops avoid pipeline stalls
  • Algorithms are increasingly designed around arithmetic intensity (FLOPs per byte of memory traffic) rather than raw operation count
  • In MPI, always ensure sends and receives are paired symmetrically. A MPI_Send with no matching MPI_Recv will block forever (or deadlock in collective operations).
  • Prefer collective operations (MPI_Bcast, MPI_Reduce) over manual point-to-point loops — they're optimized for your hardware topology and much harder to deadlock.
  • In Go: when a channel send or receive seems stuck, check whether the corresponding goroutine has already exited. A channel with nobody on the other end blocks forever.
  • Cache-friendly data structures (arrays over linked lists) matter enormously in HPC — pointer chasing across DRAM is the single biggest performance killer.
  • The message-passing model is naturally deadlock-resistant — but synchronous channels/sends in a cycle (A waits for B, B waits for A) are the classic deadlock pattern. Introduce a buffer or reorder.
